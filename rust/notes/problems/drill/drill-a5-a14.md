## a14 のバグ — Rust 初心者の定番罠

書いたコード:

```rust
let c: char = /* 入力から */;
for x in s.chars() {
    match x {
        c => ans += 1,       // ← ここで c は「変数 x の値を束縛する新しい変数」になる
        _ => {}
    }
}
```

**`match x { c => ... }` の `c` は「外側の変数 c の値と比較する」ではなく「新しいパターン変数 c を作って x を束縛する」**。
結果として全文字が第1アームに入って、`ans == s.len()` になっていた。

| パターン記法 | 意味 |
|---|---|
| `'a'` リテラル | 値マッチ（`x == 'a'` のとき） |
| `'a'..='z'` 範囲 | 値マッチ（範囲内のとき） |
| `Some(x)`, `(a, b)` | 構造分解 |
| **`c` 識別子** | **束縛**（必ずマッチして名前を付ける） ★ |
| `c @ 'a'..='z'` | 束縛 + 値マッチ |

> ✏️ **覚え方**: パターン内の **裸の名前は常に束縛**。値で比較したいなら **マッチガード `if x == c`** か **`if 文`** にする。

直し方3つ:

```rust
// 1. マッチガード（match を残したいなら）
match x {
    y if y == c => ans += 1,
    _ => {}
}

// 2. そもそも if で十分
if x == c { ans += 1; }

// 3. イテレータ版
let ans = s.chars().filter(|&x| x == c).count();
```

A-9 で同じ match パターンが動いたのは、リテラル `'a'` を書いていたから値マッチになっていた。**変数 c を直接埋め込んだ瞬間に意味が変わる** のが罠。

知識レベル: 🔴 罠を踏んだ。今度から「match に変数名を直書きしない」をルール化する。

## 苦手分野の整理（a5〜a14 から）

### ① `char` と `String/&str` の使い分けが曖昧

a6 で書いた:
```rust
let ans: String = s[..1].parse().unwrap();
```

`.parse()` は **文字列を数値などに変換するためのメソッド**。`&str → String` で使うのは目的外（動くけど）。
自然な書き方:

```rust
let ans = &s[..1];         // &str。println! で問題なし
// または
let ans = s.chars().next().unwrap();   // char
```

**`char` と `&str` と `String` の3つの世界がある**ことを意識する:

| 型 | 何 | リテラル | 取り方の例 |
|---|---|---|---|
| `char` | 1文字（4バイトの Unicode） | `'a'` | `s.chars().next()` |
| `&str` | 文字列スライス（借用） | `"abc"` | `&s[..1]` |
| `String` | 文字列（所有） | なし | `s` 自体 |

`println!("{}", x)` はどの型でもいける（Display さえあれば）。だから「とりあえず動かす」段階では型の違いを意識しなくても通る → **そこに罠**。

知識レベル: 🟡 雰囲気で使い分けてる。

### ② iterator メソッドへの抵抗

a6, a7 で `chars().next()` / `chars().last()` を使わなかった。スライスに逃げた。  
これは **「`chars()` → 文字（char）」「`iter()` → 要素」のイテレータパターン** にまだ慣れていないサイン。

イテレータの基本セットを覚える:

```rust
// 文字列 → 文字
s.chars().next()         // Option<char>
s.chars().last()         // Option<char>
s.chars().nth(2)         // Option<char>（0-indexed の3番目）
s.chars().count()        // usize（文字数）
s.chars().filter(|&c| c == 'a').count()   // 特定文字の個数
s.chars().rev().collect::<String>()        // 反転
s.chars().map(|c| c.to_uppercase().next().unwrap()).collect::<String>()  // 全部大文字

// 配列 → 要素
v.iter().min()           // Option<&T>
v.iter().max()
v.iter().sum::<i64>()
v.iter().filter(|&&x| x > 0).count()
v.iter().enumerate()     // (index, &value) のタプル
```

知識レベル: 🟡 知ってる種類は少しずつ増えてる。場面で出てこない。

### ③ `to_digit` / `as i64 - '0' as i64` がまだ手に馴染んでない

a13 で「覚えるの長い」と感じた。3パターン覚えれば、どれかが思い出せる:

```rust
// 方法1: to_digit
let n: u32 = c.to_digit(10).unwrap();

// 方法2: ASCII 差
let n: i64 = (c as i64) - ('0' as i64);

// 方法3: 文字列にして parse
let n: i64 = c.to_string().parse().unwrap();
```

**自分の好みを1つ決めて、それだけ覚える**のがコツ。3つ覚えようとすると逆に出てこない。  
個人的おすすめは方法1（`to_digit(10).unwrap()`）。数値以外のときに None が出るのも安全。

知識レベル: 🟡 ヒント見れば書ける。

### ④ 「複雑になりそう」への不安

a10 のコメントに「特定の文字だけ大文字とか言われたら死にそう」とあった。これは正直な不安。  
実は `chars().map().collect()` を知っていれば、複雑な条件付き変換も書ける:

```rust
// 例: 母音だけ大文字にする
let ans: String = s.chars()
    .map(|c| if "aeiou".contains(c) { c.to_ascii_uppercase() } else { c })
    .collect();
```

このパターン1つで「各文字を条件次第で変換」ができる。**死なないために覚えるべきは「map + collect」の組み合わせ**。

知識レベル: 🟢 知らなくて当然。次に出てきたら覚える。
