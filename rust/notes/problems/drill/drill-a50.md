## 問題

[[../../contests/drill/a50|問題文]]

長さ N の数列 A の隣り合う2要素の差 A[i+1] - A[i] の最大値を出力する。

## 実装アプローチ

**自力ループ版（最初に書いたやつ）**:
```rust
use proconio::input;

fn main() {
    input! {
        n: usize,
        a: [i64; n],
    }
    let mut diff: Vec<i64> = vec![0i64; n-1];
    for i in 0..n-1 {
        diff[i] = a[i+1] - a[i];
    }
    let max = *diff.iter().max().unwrap();
    println!("{}", max);
}
```

**`windows(2)` 版（推奨）**:
```rust
let max = a.windows(2).map(|w| w[1] - w[0]).max().unwrap();
println!("{}", max);
```

最初 `for i in 0..n` で書いて `a[n]` の範囲外アクセスでテストが落ちた → 自力で `0..n-1` に修正。**自力デバッグできたのが収穫**。

## 学んだこと

### `.windows(k)` — 連続 k 要素のスライスを返すイテレータ

```rust
let a = vec![1, 2, 3, 4, 5];
for w in a.windows(2) {
    println!("{:?}", w);   // [1,2], [2,3], [3,4], [4,5]
}
```

- `a.windows(k)` は **連続する k 個** のスライス `&[T]` を順に返す
- 個数は **`N - k + 1`** 個（k=2 なら N-1 個）
- 元の Vec を壊さない（借りるだけ）
- スライスなので `w[0], w[1], ..., w[k-1]` でアクセス可能

### 「境界処理を消せる」のがポイント

```rust
// ループ版: n-1 までで止める必要あり、境界バグの温床
for i in 0..n-1 {
    let d = a[i+1] - a[i];
    ...
}

// windows 版: そういう境界処理が消える
for w in a.windows(2) {
    let d = w[1] - w[0];
    ...
}
```

「`n-1` か `n` か」「`< n-1` か `<= n-1` か」を考えなくていい。[[drill-a48]] の番兵テクと同じ「境界をフレームワークに押し付ける」思想。

### `windows` の使いどころ — 連続 k 要素を扱う問題

| やりたいこと | 書き方 |
|---|---|
| 隣接差の最大 | `a.windows(2).map(\|w\| w[1]-w[0]).max()` |
| 連続2要素の積の最大 | `a.windows(2).map(\|w\| w[0]*w[1]).max()` |
| 連続3要素の和 | `a.windows(3).map(\|w\| w.iter().sum::<i64>())` |
| K日連続の最大利益 | `a.windows(k).map(...).max()` |
| 単調増加判定 | `a.windows(2).all(\|w\| w[0] < w[1])` |

連続 k 要素を扱う問題は反射で `windows(k)` を出せるようにしたい。

知識レベル: 🟢 実装可能

### 関連: `chunks(k)` との違い

| メソッド | 動作 | 個数 |
|---|---|---|
| `a.windows(k)` | **重なりあり** で連続 k 個 | N - k + 1 |
| `a.chunks(k)` | **重ならず** k 個ずつ区切る（最後は短いことも） | ⌈N / k⌉ |
| `a.chunks_exact(k)` | 重ならず k 個ずつ（最後の半端は無視） | ⌊N / k⌋ |

例: `[1,2,3,4,5].windows(2)` → `[1,2],[2,3],[3,4],[4,5]`（4個、重なる）
例: `[1,2,3,4,5].chunks(2)` → `[1,2],[3,4],[5]`（3個、重ならない）

「スライディングウィンドウ系」は `windows`、「ブロック処理」は `chunks`。

### イニシャライズで `vec![0; n]` した後の `+=` vs `=`

```rust
let mut diff = vec![0i64; n-1];
diff[i] += a[i+1] - a[i];   // 初期値0なので += でも = と同じ
diff[i] = a[i+1] - a[i];    // こちらが意図に合う
```

- 「あとから加算していく」なら `+=`（出現回数カウントなど）
- 「1回だけ代入する」なら `=`

意図が「1個ずつ書き込む」なら `=` のほうが読み手に親切。意味は同じだが「ここで加算する意図はない」が伝わる。

知識レベル: 🟡 雰囲気理解

### 自力デバッグできたこと

「`for i in 0..n` で動かない → テストで気付く → `0..n-1` に直す」が**自力でできた**のは大きい。

範囲外アクセス（panic: index out of bounds）は Rust が実行時に守ってくれるので、テストで落ちる場所がわかる。`cct aXX` で 1ケースでも落ちたら範囲・境界を疑う癖をつける。

知識レベル: 🟢 実装可能
