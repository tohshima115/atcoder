## 問題

[[../../../contests/drill/b4|問題文]]

$H$ 行 $W$ 列の整数行列を読み込み、**各行の合計**を出力する。

## 実装アプローチ

**提出コード**:
```rust
use proconio::input;

fn main() {
    input! {
        h: usize,
        w: usize,
        v: [[i64; w]; h]
    }
    for i in 0..h {
        let ans = v[i].iter().sum::<i64>();
        println!("{}", ans);
    }
}
```

方針：proconio に 2 次元配列をそのまま読ませ、各行を `v[i].iter().sum()` で合計。**O(H·W)**。

> このセッションのテーマは「短く書けすぎて逆に不安」→ なぜこれで正しいのか、そして『列合計』に変えると何が変わるのかを掘り下げる回。

---

## 学んだこと

### proconio の 2 次元配列 `[[i64; w]; h]` の読み方

**内側から外側へ**読む:

- `[i64; w]` … `i64` が `w` 個 → 1 行ぶん（`Vec<i64>`）
- `[[i64; w]; h]` … その行が `h` 個 → `Vec<Vec<i64>>`

proconio が `H·W` 個の数値を読んで、勝手に `H` 行 × `W` 列に詰めてくれる。**自分で二重ループして 1 個ずつ読む必要がない**のがミソ。C の `scanf` 二重ループの感覚だと「短すぎて不安」になるが、これが普通。

- `v[i]` … i 行まるごと（`Vec<i64>`、長さ `w`）
- `v[i][j]` … i 行 j 列の 1 要素

知識レベル: 🟢 実装可能

---

### `sum::<i64>()` のターボフィッシュ（型注釈）

`sum()` は「何の型に合計するか」を推論できないと怒る。`::<i64>` で「`i64` に合計せよ」と明示している。`v` が `i64` なのでこれで合う。`usize` などにすると型が合わずエラー。

知識レベル: 🟢 実装可能

---

### `(0..w)` は `Range` 型で「最初からイテレータ」

行合計はこう書ける:
```rust
v[i].iter().sum::<i64>()           // ① Vec を iter() でほどいて足す
(0..w).map(|j| v[i][j]).sum()      // ② 0..w から添字を作って v[i][j] を拾って足す
```

`0..w` は配列ではなく **`Range { start: 0, end: w }` という構造体**。そして `Range` は**生まれつきイテレータ**で、`0,1,…,w-1` を順に吐く。だから `for` にもそのまま回せるし、`.map()` `.sum()` も直接生やせる。

**なぜ `.iter()` が要る／要らないか**:

| 書き方 | 正体 | イテレータにするには |
|---|---|---|
| `v[i]` | `Vec<i64>`（コレクション） | `.iter()` が必要 |
| `0..w` | `Range`（もうイテレータ） | そのままで OK |

`.iter()` は「Vec → イテレータ」の橋渡し。`Range` はもう向こう岸にいるので橋が要らない。

> なお `v[i].iter()` が吐くのは中身への**参照** `&i64`。`sum()` は `&i64` でも `i64` でも足せるよう作られている（b1〜b2 でやった filter の参照剥がしと同じ筋）。

知識レベル: 🟢 実装可能

---

### 列合計に変えると「ひと手間」増える理由

行合計が一瞬なのは、`v[i]` が**行まるごとの塊**として取れるから。一方、列は `v[0][j], v[1][j], v[2][j]…` と**バラバラの行から 1 個ずつ拾い集める**必要がある（`v[j]` では取れない＝それは j 行目）。

**方法A：ループの内外を入れ替えるだけ**
```rust
for j in 0..w {                          // 外: 列
    let ans = (0..h).map(|i| v[i][j]).sum::<i64>();  // 内: 行を縦に舐める
    println!("{}", ans);
}
```
`v[i][j]` の `i` を動かして固定の `j` を縦に足す。行合計と外/内をひっくり返しただけ。

**方法B：列ごとの合計箱に足し込む（1 回走査）**
```rust
let mut col = vec![0i64; w];
for i in 0..h {
    for j in 0..w {
        col[j] += v[i][j];               // j 番目の箱に足し込む
    }
}
```

知識レベル: 🟢 実装可能

---

### 行合計と列合計：オーダーは同じ、定数倍で差が出る

足し算の回数を数えると:

- 行合計：`w` 回 × `h` 行 = `H·W`
- 列合計：`h` 回 × `w` 列 = `W·H`

**全マスをちょうど 1 回ずつ触る**ので、計算量はどちらも **O(H·W) で同じ**。「列合計だから TLE」という心配は基本不要。

差が出るのは**定数倍＝メモリの触り方**。`Vec<Vec<i64>>` は各行がメモリ上で連続して並ぶ:
```
行0: [a b c d]   ← 連続
行1: [e f g h]
行2: [i j k l]
```
- **行合計**は `a→b→c→d` と隣を順に読む → キャッシュが効いて速い。
- **列合計（方法A）**は `a→e→i`（次の行へジャンプ）と飛び飛びに読む → キャッシュミスで同じ回数でも遅くなりがち。
- **方法B**は `v` を行順に素直に舐めながら集計するので、列合計なのにメモリアクセスは行に優しい → 列合計を**キャッシュ効率よくやるなら方法 B が一枚上手**。

> これは「列方向の集計はメモリの並びに逆らうので不利」という、**列指向 DB（BigQuery など）が解こうとしている問題と同じ構図**。「メモリ/ストレージ上の並び」と「集計したい方向」が一致しているかが効く。

**競プロ的結論**: まず素直に書いて OK。H,W が数千なら誤差。「同じオーダーなのに片方だけ妙に遅い」場面で「並びに逆らってるかも」と思い出せると一段上。

知識レベル: 🟡 雰囲気理解

## このセッションの総括

| 観点 | 学び |
|---|---|
| proconio `[[i64; w]; h]` | 内側から読む。2 次元入力を自動で詰めてくれる |
